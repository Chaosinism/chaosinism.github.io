音乐编程工具考察

  2020-04-16   经验心得   MIDI VST 软件脚本 资源汇总  评论 字数统计: 3476字 阅读时长: 7分

作为没有艺术细胞又偶尔想搞创作的人，我希望能用程式化的方式处理各种媒体。对于图像或视频，现在的主流创作工具都提供了脚本编程的功能，也有各种经典或前沿的研究成果被做成开源的库以便直接取用。但这种做法到了音乐好像行不通了——音乐看起来能用很数学的方式来描述，但恐怕这只是一种错觉。从科学史中可以看到大量数学家与物理学家试图破解音乐的奥秘，然而至今不论是处理、分析还是创作音乐的技术和理论都远未成熟，更不用提实际应用到生产环境。因此在进行某些无谓的尝试之前，我需要对现有的工具先调查一番，不然甚至不知道该从哪里开始。

正文之前，还要说明几点。首先，与本栏目的其它文章不同，这篇文章不包含什么我的个人成果，我对此领域的了解处在一个很初级的阶段，做这些调查更多是为了整理思路，同时也不知道其中是否有错漏。其次，我自觉与不自觉地总是拿图形学或图像处理技术的观点来类比音频处理，不知道这种看法会不会冒犯到什么人。因此还望读者自行判断以下内容的价值。

音乐编程的挑战

以我当前浅薄的认知来看，量化处理音乐最棘手的困难是如下几条：

  • 音乐常用的量化表达方式有声音波形和音符序列，两者的处理方式迥然不同，缺少通用的方法，也难以互相转化（由多声部音频中扒谱是音乐人的基本技能，而机器至今无法完美实现）。同时，两者中无论哪个都缺乏表征更高层次的属性的方法，如整首乐曲的段落结构与风格（以图像处理作类比，可使用Gram矩阵表征图像风格；而音乐的机器风格迁移还没有较好的成果，即使这个任务对人类音乐人并不难）。
  • 现行的音乐处理技术与标准非常重视实时性。在图像领域，即时与非即时渲染都发展出多套算法，而音乐领域则将大部分精力放在前者，历史上一些复杂算法（如卷积混响）即使效果好也得不到重视，直到硬件进化到能实时运行它为止。这导致一些很简单的操作却没法顺畅进行，比如我早先奇怪为什么许多作者用非线性剪辑软件而不是DAW来做音MAD，后来发现原因之一可能是MIDI+VST的流程中难以实现按音符时长来拉伸/压缩采样——采样器看到一个音符开始就会播放音频，此时它根本不知道这个音符有多长。
  • 也许是上面两条导致的结果，对音乐的抽象表达/编程语言并不是太少，反而是太多。可想而知，工业界难以兼容如此多的标准。坦白地说，虽然这些语言宣称了各种各样的优点，我完全不知道怎么将它们方便地整合进实际的音乐制作流程中。既然我不想跑到实验室里做音乐，也不是要写论文，就不太可能从上述列表中受益。因此我更想从实用性出发来做一份自己的整理。

考虑到以上几点，这篇文章将以音符序列/声音波形和实时/非实时演算的分类为线索，寻找并整理一些尽可能与主流音乐制作流程（如DAW+MIDI+VST）兼容的编程工具。（部分工具适用于多个分类，我按照其强项来划分。）

音符序列 + 实时

标准

  • MIDI

绝大多数电子乐器使用MIDI信号，将音符、音高、按键力度等属性传递给宿主。虽然MIDI已有30多年历史，很多特性看起来很老旧，但并没有什么别的选择。

工具

  • Serpent

卡内基梅隆大学开设一门音乐编程处理的课程，其中用到Serpent语言。这个语言专门针对实时系统而设计，并含有处理MIDI信号的模块。不过课程基本未涉及商业软件，因此不知道它在实际音乐制作流程中的效果如何。

音符序列 + 非实时

标准

  • MIDI

对于非实时的音符序列，最通用的表示形式似乎仍然是MIDI，比如将乐谱保存为mid格式的文件。但此时MIDI针对硬件特性的各种考虑不再起作用，反而变得更像一种累赘。另外，能够读取实时MIDI信号的工具未必能够解析mid文件，反之亦然，这一点也很不方便。

  • DAW的脚本支持

那么各种DAW针对这些问题是否有解决方案呢？Cakewalk、Ableton与Reaper都有自己的一套脚本语言来操作音符序列，但互相并不兼容，另一些DAW则没有这样的功能。

  • MusicXML

一种流行的用于记录乐谱的格式，虽然主流DAW很多都能读取或输出它，但还是主要用于乐谱的流通，在创作或音频处理中发挥的作用不及MIDI。

  • Scala

记录调式的文件格式，部分VST可以读取，详见下面的工具介绍。

工具

  • NAudio (https://github.com/naudio/NAudio)

这是C#编程时可以用的库，我在编写Vegas脚本时经常用到。它提供的API能以很直观的方式解析mid文件，对于主流DAW所生成的mid都是可以正确读取的，也可以写入新的文件。另外虽然我自己没有实践过，但它也能实时读取和发送MIDI信号。

这个库也有很多针对声音波形的函数，不过大都是音频转码一类的功能，不属于本文讨论的范畴，所以后文就不多谈了。

  • music21 (https://web.mit.edu/music21)

这是较有名的一个Python库，它不仅能读取MIDI和MusicXML文件，更重要的是包含了很多乐理的内容，如音调与和弦分析等。它看起来也挺难与DAW互动的，或许支持Python的Reaper可以吧。

  • Magenta Studio (https://magenta.tensorflow.org/)

它是Google针对音乐领域发起的机器学习项目，目前已经有续写歌曲、生成鼓点等多个功能，既可以独立使用也可以作为Ableton的插件。我个人试用后感觉它现在生成出的东西质量并不算高，但这恐怕已经是同类项目中最接近实用的之一了，也只能继续期待。它包含的神经网络模型开放给所有人，可以自行用MIDI文件训练。

  • Scala (http://www.huygens-fokker.org/scala/)

到这里为止，本文提到的音符序列都是特指西方调式和十二平均律，所以上述工具牵扯到微分音或纯律时就不好用了。Scala在这方面提供了很全面的功能，包括多种分析工具、命令行交互与输入输出。调式的探索几乎是一个完全不同的领域，涉及大量知识，要另找时间研究才行，只是这里也有必要列举一下。

声音波形 + 实时

标准

  • VST

VST是当前最著名的音频插件标准，由Steinberg公司开发，同时支持MIDI与声音波形作为输入输出，因此既可制作乐器又可制作效果器。其它类似的标准还有用于MAC的AU和用于Linux的LADSPA等。

  • JSFX

开发VST需要使用Steinberg专门的SDK，每次修改都要编译也较为繁琐。我希望找到一个脚本语言以便于快速开发，目前较好的选择应该是JSFX，虽然文档比较简陋，但看起来必要的功能都有提供，也可以在网上找到不少现成的脚本。

工具

  • ReaJS (https://www.reaper.fm/reaplugs/)

REAPER对JSFX的支持最为完善，搭载了可以运行脚本的内置插件ReaJS，同时此插件也可以在其它DAW中使用，可以说是做了一件大好事。我如果进行编程尝试，可能首先考虑使用它。官方资料较少，好在也有其它音乐人\程序员在积极钻研它，比如这位Robert Randolph的个人网站。

  • Minihost Modular (http://support.image-line.com/redirect/Minihost_Modular_Beta)

我关心的另一项技术是节点编辑器，近年来视频特效、3D与游戏引擎都开始应用这种方式，它将一些较简单的函数作为节点，允许操作者将它们按不同的方式连接起来，这样不需写代码就可以实现复杂的逻辑。我试图寻找音乐制作是否有类似工具，目前看到的比较好的选择是Minihost Modular。它是为FL Studio开发的插件，但也可以作为VST被其它DAW使用，它可以将任意其它VST作为节点，把它们的输入和输出互相连接。但是VST的参数并不能也用相同的方式连接，因此它不如视频或3D软件的节点编辑器那样灵活。

（2020/05/30更新：Carla也有类似功能，优缺点有待考察。）

  • SuperCollider (https://supercollider.github.io/)

一个人气相当高的项目，看起来是可编程的声音合成引擎。说实话我并没有弄明白它在同类产品中脱颖而出的原因是什么，但至少其具有非常完善的功能与文档，并经过了长期的开发与改进，或许值得继续关注。

声音波形 + 非实时

标准

  • Nyquist

声音波形的非实时处理类比于图像大概就好像Photoshop的滤镜，就像图像处理算法可以直接作用于像素，那声音处理算法也可以直接作用于wav等格式。原始的波形文件并没什么可介绍的，编码和压缩技术也不属于本文的讨论范围，本文更关心是否有存在更高级的抽象来更方便地处理音频，但没找到完全符合我要求的。Audacity是最著名的开源音频编辑软件，它提供一种Nyquist语言来进行可编程的声音合成与分析。由于它基于Lisp，学习成本恐怕不低。

工具

  • aubio (https://aubio.org/)

用于分析音乐的Python库，提供快速傅里叶变换、音调/节奏检测、滤波等函数，都是很实用的功能，并且兼容NumPy数组的格式，看起来是一个不错的工具，这些功能的效果如何我暂时还没有测试过。

  • WaveNet (https://github.com/ibab/tensorflow-wavenet)

可能是目前最具代表性的处理音频的神经网络，虽然更多地被用于语音领域，也有人用它研究音乐分类、合成与风格迁移等。由于深度学习所需的巨大计算量，我将它列到非实时处理的这一栏，但其实它遵循的逻辑还是类似实时处理——采用“因果卷积”由过去的数据推测未来的数据，我也没什么专业知识来分析这种做法的优缺点，只是感觉它生成出来的东西似乎还有所欠缺。
